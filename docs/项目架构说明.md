# 项目架构说明

## 技术栈

项目保持原课程设计技术栈，不引入 Express、NestJS、Element Plus、Ant Design 等大型框架。

| 层级 | 技术 |
| --- | --- |
| 前端 | Vue 3、Vite、Pinia、Vue Router、原生 CSS |
| 后端 | Node.js 原生 `http` 服务、模块化路由、JWT |
| 数据库 | MySQL 8，开发环境可使用自动初始化演示数据 |
| 临时验证数据 | 开发环境内存 `Map`，生产环境 Redis |
| 测试 | 后端接口冒烟测试、Playwright E2E |

## 分层结构

```text
backend/
  index.js                 # 原生 http 服务入口
  src/admin-routes.js      # 后台管理接口
  src/client-routes.js     # 前台用户接口
  src/modules/             # 订单、支付、会员、社区、看板等业务模块
  src/shared/              # 鉴权、响应、校验、MySQL、验证码存储等通用能力

front/
  src/router/              # Vue Router 页面路由
  src/stores/              # Pinia 状态管理
  src/views/               # 前台页面
  src/AdminApp.vue         # 独立后台管理页面
  src/components/          # BaseButton/BaseCard/BaseModal/BaseTable/BaseFormItem 等组件
  src/styles.css           # 原生 CSS 视觉体系和响应式样式

scripts/
  backend-api-smoke.test.js # 后端接口冒烟测试
  e2e.js                    # E2E 服务启动与清理脚本

tests/e2e/
  coffee-book.spec.js       # Playwright 关键业务流程
```

## 核心业务流

1. 用户登录后在商品页加入购物车，提交订单。
2. 订单创建后状态为 `pending_payment`，前端跳转 `/pay/:orderId`。
3. 支付页创建 `payments` 记录，展示模拟二维码和 15 分钟倒计时。
4. 用户点击“我已支付”，支付状态变为 `submitted`，订单状态变为 `payment_review`。
5. 管理员进入 `/admin/payments`，确认收款后支付状态变为 `confirmed`，订单状态变为 `paid`。
6. 用户订单详情和会员中心通过接口重新拉取数据，展示“已支付”。

## 后台看板

后台 `/admin.html` 内的数据看板来自 `GET /api/admin/summary` 的 `dashboard` 字段。后端聚合订单、预约、用户、商品、订单明细数据，前端用原生 CSS 绘制指标卡、条形图和表格排行，避免引入大型图表库。

## 答辩说明要点

- 项目没有重构为全新项目，而是在原有路由、模块和页面结构上增强。
- 前后台共用统一视觉组件和响应式 CSS，但后台保留独立入口 `/admin.html`。
- 临时安全数据使用存储抽象，开发环境便于演示，生产环境可切换 Redis。
- 自动化测试会创建带批次标识的数据，并在结束后调用后台接口清理，不污染演示数据。
