# 验证码与滑块验证实现说明

## 设计目标

验证码模块用于降低短信验证码被批量刷取、滑块固定值绕过、验证码重复使用等风险。实现位置为 `backend/src/shared/verification-store.js`，前台登录页在 `front/src/views/Login.vue` 中提交滑块 token、滑动坐标和时间戳。

## 存储策略

| 环境 | 存储方式 | 说明 |
| --- | --- | --- |
| 开发环境 | 内存 `Map` | 便于本地课程演示，不需要额外服务 |
| 生产环境 | Redis | 配置 `REDIS_URL` 后启用，适合多进程部署和自动过期 |

如果 `NODE_ENV=production` 但未配置 `REDIS_URL`，后端会拒绝启动，避免生产环境继续使用进程内存保存安全数据。

## 图形验证码

- 生成随机 token 和 4 位数字答案。
- 服务端只保存答案、签发时间、过期时间、使用次数、失败次数。
- 校验成功后立即删除，防止重复使用。
- 校验失败会增加失败次数，超过限制后删除 challenge。

## 短信验证码

- 验证码有效期为 5 分钟。
- 同一手机号有发送冷却时间，避免高频请求。
- 验证码不返回给前端，开发环境只打印在后端终端，格式为 `[dev sms] 13800000000: 123456`。
- 校验成功后删除，错误次数达到限制后删除。

## 滑块验证

滑块 challenge 会保存随机目标坐标 `target` 和纵向干扰值 `y`，校验时同时检查：

- token 必须存在且未过期。
- 用户提交的 x 坐标必须在 0 到 100 范围内。
- x 坐标和服务端随机 target 的误差必须在允许范围内。
- 前端提交 `sliderStartedAt` 和 `sliderEndedAt`。
- 客户端滑动耗时不能过短，服务端签发到校验也不能过短。
- 时间戳不能早于 challenge 签发时间，也不能明显晚于服务器当前时间。

这样可以避免只提交固定坐标值绕过滑块，也能降低脚本瞬时提交的风险。

## 答辩说明

本模块的核心不是“做真实短信服务”，而是展示完整的验证码安全设计：临时数据不落库、生产环境可 Redis 化、challenge 有过期和次数限制、验证成功后一次性消费、失败后逐步限制。
